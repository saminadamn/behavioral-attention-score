Can a learned policy replace the rule engine?

The deployed system (see the Simulation Demo and Dashboard) plans interventions with a transparent rule-based engine — every decision comes with a written reason. This page covers a separate, experimental question: what happens if we train a neural network to make that decision instead? Short answer: it runs, it's real, and it also reveals exactly why replacing the rule engine today would be premature.

Why "offline" learning?

There's no live classroom to practice on here — just one dataset, already generated and logged by the rule engine. So instead of learning by trial-and-error in real time, the network studies the rule engine's own past decisions, like a student reviewing recorded lessons instead of attending them live.

Why keep the rule engine as default?

The network only ever sees what the rule engine already chose to do. It has almost no information about actions the rule engine rarely tried — so trusting it over the transparent, reason-giving rule engine today would be a real risk, not a caution being overly careful.

Four techniques, explained simply

Double DQN

One part of the network picks the best-looking next move; a second, steadier copy checks how good that move actually is. Splitting the two jobs stops the network from fooling itself into overconfidence.

Prioritized Experience Replay

Instead of practicing on random past examples equally, the network revisits the ones it's currently getting most wrong more often — like re-studying the questions you keep missing.

LSTM Memory

Reads a short window of recent interactions instead of one in isolation, so it can pick up on a trend building over several turns.

Conservative / Constrained Learning

CQL, IQL, and BCQ are three add-ons that make the network deliberately cautious about actions it has little evidence for — instead of confidently guessing, it stays close to what the logged data actually supports.

What actually happened when we trained it

Model Stays within the logged policy's behavior? Agreement with logged decisions
Rule Engine (deployed) — reference — 100%
Vanilla DQN Extrapolates away from it ~3%
Double DQN + PER + LSTM (DRQN) Extrapolates away from it ~4%
Conservative Q-Learning (CQL) Stays close by design ~86%
Constrained BCQ Stays close by design ~87%

The story here isn't "CQL and BCQ perform better." It's that a plain offline network is free to extrapolate to actions it barely saw in the logs — so it drifts far from what the rule engine actually did. CQL and BCQ are specifically built to stay close to the logged behavior policy instead of extrapolating, and their much higher agreement is evidence that constraint is working — not proof that a conservative network would make better decisions than the rule engine.

What this page is not claiming

  • Not a replacement for the deployed rule engine — that stays the system of record.
  • Not validated against any real student outcome — the reward is a designed proxy, not measured learning benefit.
  • "Agreement" measures similarity to logged decisions, not correctness — there's no ground truth for what the ideal decision was.
  • Not trained or evaluated on real classroom data — everything here is synthetic, exactly like the rest of this project.
